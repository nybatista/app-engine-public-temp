## ğŸŒ Hosting & Deployment

SpyneJS applications are **single-page applications (SPAs)** and can be hosted on any modern static or server-backed platform.

There is no platform lock-in. The only requirement is correct routing behavior.

### Core Hosting Rules

1. **All application routes must resolve to `index.html`**
    - Any request to a path that is not a real file should return `index.html`
    - This allows the SpyneJS runtime to handle routing internally

2. **Static assets must be served directly**
    - Requests to `/assets/*` should return the corresponding file
    - These requests should NOT be redirected to `index.html`

In short:
- **App routes â†’ `index.html`**
- **Asset routes â†’ actual files**

---

### Universal Hosting Prompt

You can use the following prompt when configuring or asking for help deploying this app on any platform:

> This is a single-page application.
>
> All requests to application paths should return `index.html`.
>
> Requests for static resources under `/assets/*` should return the corresponding asset directly and must not be rewritten.
>
> The server should not perform any client-side routing logic beyond this rewrite behavior.

This configuration works for platforms such as:
- Vercel
- Netlify
- CloudFront
- S3
- Nginx
- Apache
- Any CDN or static host with rewrite support

---

### Embedding in Other Applications

SpyneJS supports embedding inside other applications (CMSs, portals, shell apps).

When embedding:
- Use the **`query` route type** to drive application state via URL parameters
- Avoid relying on path-based routing when SpyneJS is not the top-level app

This allows SpyneJS to coexist cleanly inside existing systems without interfering with host routing.
